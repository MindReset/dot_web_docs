Quote/0 のコンテンツ更新は、固定されたアラームのように動作するものではありません。デバイスは現在時刻、給電状態、コンテンツモード、スリープ状態、サーバー側のスケジューリング結果をもとに、次回の更新時刻を動的に計算します。

API を使ってより正確なタイミングで内容を表示したい場合は、Quote/0 を継続して電源に接続しておくことをおすすめします。

## 更新時刻の計算方法

<Steps>
<Step>

### デバイスが電源に接続されているか確認する

電源に接続されている場合、デバイスはより安定してネットワーク接続を維持できます。デスクトップのステータス表示、オートメーションの通知表示、家庭内の情報端末など、常時オンラインに近い情報表示に適しています。

バッテリー駆動時は、デバイスはバッテリー持続時間を優先します。コンテンツ更新が不要なときは低消費電力のスリープに入り、ネットワーク接続を継続的には維持しません。

</Step>

<Step>

### デバイスがスリープ時間帯にあるか判断する

デバイスがスリープ時間帯にある場合、システムは通常のループ間隔で繰り返し起動するのではなく、スリープ終了を優先します。

デバイスがスリープ中に API で新しいテキストや画像コンテンツを設定した場合、その内容はまずサーバーに保存されます。デバイスは次回の自動起動後、Wi-Fi とサーバーへ再接続してから最新内容を取得し、表示します。

</Step>

<Step>

### コンテンツモードから更新候補時刻を計算する

デバイスは複数のコンテンツモードを組み合わせて、次に更新できる時刻を計算します。

- ループコンテンツ：ループリストと更新間隔から次の切り替え時刻を計算します。
- 固定コンテンツ：固定コンテンツの開始時刻と終了時刻から、切り替えが必要かを判断します。
- スリープ設定：スリープの開始時刻と終了時刻から、更新を延期するかを判断します。
- 日付の切り替わり：一部の場面では、日付をまたぐタイミングも再計算に関わります。

複数の時刻候補がある場合、システムはより適切な早い時刻を選んで更新します。

</Step>

<Step>

### 前回の実際の更新時刻を新しい基準にする

次回の自動更新は、通常、前回の実際の更新時刻を新しい基準として計算されます。毎時 0 分や 30 分に固定されるわけではありません。

たとえば、更新間隔を 30 分に設定している場合：

| 操作 | 次回の更新時刻 |
| --- | --- |
| 5:45 に自動更新 | およそ 6:15 に再更新 |
| 6:10 に手動更新 | およそ 6:40 に再更新 |

手動更新、Dot. アプリ内でのコンテンツ切り替え、API による即時切り替えは、いずれも次回自動更新の計算基準を変える可能性があります。

</Step>

<Step>

### サーバー側で必要なスケジューリングの揺らぎを持たせる

多数のデバイスが同じ時刻にサーバーへ集中してリクエストすることを避けるため、システムは必要な範囲で時刻に揺らぎを持たせます。これにより、全体の安定性を高め、リクエストの滞留やタイムアウトを減らします。

そのため、Quote/0 の自動更新は周期的な情報表示に適しており、厳密な定時トリガーとして使う用途には向いていません。

</Step>
</Steps>

## 電源接続とバッテリー駆動の違い

電源に接続されている場合、Quote/0 はより安定してネットワーク接続を維持でき、常時オンラインに近い情報端末として使いやすくなります。この状態で API により内容を設定し、`refreshNow` を `true` にすると、スクリプト実行、オートメーション通知、リアルタイムの状態表示に適しています。

バッテリー駆動時は、Quote/0 はバッテリー持続時間を優先します。スリープ中はネットワーク接続を継続的には維持しません。API で送信した内容はまずサーバーに保存され、デバイスが次回自動起動したあとに取得・表示されます。

## 固定コンテンツとループコンテンツが更新に与える影響

ループコンテンツは、リストの順序と設定した間隔に従って切り替わります。

固定コンテンツは、指定された時間帯に有効になります。固定コンテンツの時間帯に入るとき、または時間帯から出るときは、通常のループ間隔の終了を待たずにコンテンツが切り替わる場合があります。

デバイスがスリープ、固定コンテンツの切り替え、ループ間隔など複数の時刻候補に同時に関係する場合、システムはそれらを総合して、より適切な更新タイミングを選択します。

## よくあるシナリオ

### シナリオ 1：デバイスが電源に接続され、API で即時表示する

デバイスが電源に接続され、ネットワーク接続も安定しています。この状態でテキスト API または画像 API から内容を送信し、`refreshNow` を `true` に設定します。

この場合、デバイスは通常、サーバーからの更新タスクを比較的すばやく受け取り、新しい内容へ切り替えられます。認証コード、予定リマインダー、オートメーションスクリプトの結果、デスクトップのステータス表示など、表示タイミングが重要な用途に適しています。

### シナリオ 2：デバイスがバッテリー駆動で、スリープ中である

バッテリー駆動時、デバイスはコンテンツ更新が不要なときに低消費電力のスリープへ入ります。この状態で API から新しいテキスト内容を送信します。

その内容はまずサーバーに保存されます。デバイスはこの API リクエストによって直ちにオンライン状態を維持するのではなく、次回の自動起動後に Wi-Fi とサーバーへ再接続し、最新内容を取得して表示します。

そのため、バッテリー駆動時は、固定コンテンツ、ループコンテンツ、天気、RSS、カレンダー概要など、厳密な表示時刻を必要としない内容に適しています。

### シナリオ 3：デバイスが 30 分ごとにループ更新する

ループ更新間隔を 30 分に設定し、デバイスが 5:45 にコンテンツ更新を完了したとします。

その後ほかの操作がなければ、次回の自動更新はおよそ 6:15 に行われます。

もし 6:10 に Dot. アプリで手動切り替えを行った場合、または API で即時切り替えを行った場合、その 6:10 の更新が新しい時間基準になります。次回の自動更新は、元の 6:15 ではなく、およそ 6:40 になります。

### シナリオ 4：デバイスに固定コンテンツの時間帯が設定されている

毎日 9:00 から 10:00 まで表示する固定コンテンツを設定しているとします。デバイスは現在ループコンテンツを表示しており、ループ更新間隔はまだ終了していません。

9:00 が近づくと、固定コンテンツの開始時刻がより重要な更新候補になります。デバイスは現在のループ間隔の終了を待たずに、固定コンテンツへ優先的に切り替わる場合があります。

固定コンテンツの時間帯が終了すると、デバイスはその後の更新時刻を再計算し、適切な非固定コンテンツのロジックへ戻ります。

### シナリオ 5：多くのデバイスが同じ時刻に更新する

多くのデバイスが同じ時刻に更新するよう設定されている場合、サーバーは必要なスケジューリングの揺らぎを持たせ、同時リクエストによる滞留やタイムアウトを避けます。

そのため、デバイスが電源に接続されている場合でも、自動更新を秒単位で厳密なタイマーとして理解するべきではありません。より正確な時刻に近づけて表示したい内容では、電源に接続したデバイスを使い、API で能動的に表示を制御することをおすすめします。

## API 利用時の推奨設定

認証コード、予定リマインダー、オートメーションスクリプトの結果、リアルタイムの状態表示など、表示タイミングが重要な内容では、次の設定をおすすめします。

1. デバイスを継続して電源に接続する。
2. デバイスのネットワークを安定させる。
3. API で内容を設定する。
4. `refreshNow` を `true` に設定する。

デバイスをバッテリーで使用する場合は、固定コンテンツ、ループコンテンツ、天気、RSS、カレンダー概要など、表示時刻にあまり敏感ではない内容の表示をおすすめします。
