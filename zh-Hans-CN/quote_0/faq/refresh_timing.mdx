Quote/0 的内容刷新不是固定闹钟机制。设备会根据当前时间、供电状态、内容模式、睡眠状态和服务器调度结果，动态计算下一次刷新时间。

如果您希望通过 API 在较准确的时间显示内容，建议让 Quote/0 持续接入电源。

## 刷新时间如何计算

<Steps>
<Step>

### 确认设备当前是否接入电源

接入电源时，设备可以更稳定地保持网络连接，更适合持续在线的信息展示场景，例如桌面状态屏、自动化提醒屏或家庭信息终端。

电池供电时，设备会优先保证续航。设备在不需要刷新内容时会进入低功耗睡眠，不会持续保持网络连接。

</Step>

<Step>

### 判断设备是否处于睡眠窗口

当设备处于睡眠窗口时，系统会优先等待睡眠结束，而不是按照普通循环间隔持续唤醒设备。

如果您在设备睡眠期间通过 API 设置了新的文本或图片内容，内容会先保存到服务器。设备会在下一次自动唤醒、重新连接 Wi-Fi 和服务器后，再获取并显示最新内容。

</Step>

<Step>

### 根据内容模式计算刷新候选时间

设备会综合不同内容模式计算下一次可能刷新的时间点：

- 循环内容：根据循环列表和刷新间隔计算下一次切换时间；
- 固定内容：根据固定内容的开始和结束时间判断是否需要切换；
- 睡眠设置：根据睡眠开始和结束时间判断是否延后刷新；
- 午夜边界：在部分场景中，跨天时间也可能参与重新计算。

如果多个时间点同时存在，系统会选择更合适的较早时间点执行刷新。

</Step>

<Step>

### 以上一次实际刷新作为新的时间基准

下一次自动刷新通常会以上一次实际刷新时间作为新的基准，而不是固定在整点或半点。

例如您设置每 30 分钟刷新一次：

| 操作 | 下一次刷新时间 |
| --- | --- |
| 5:45 自动刷新 | 大约 6:15 再次刷新 |
| 6:10 手动刷新 | 大约 6:40 再次刷新 |

手动刷新、App 内切换内容、API 立即切换内容，都可能改变下一次自动刷新的计算基准。

</Step>

<Step>

### 服务器会保留必要的调度浮动

为了避免大量设备在同一时间集中请求服务器，系统会保留一定的时间浮动。这样可以提升整体稳定性，减少请求阻塞或超时。

因此，Quote/0 的自动刷新更适合周期性信息展示，而不适合作为严格的定时触发器使用。

</Step>
</Steps>

## 接入电源与电池供电的差异

接入电源时，Quote/0 可以更稳定地保持网络连接，适合作为持续在线的信息终端使用。此时使用 API 设置内容，并将 `refreshNow` 设置为 `true`，更适合实现脚本触发、自动化提醒和实时状态展示。

电池供电时，Quote/0 会优先保证续航。设备在睡眠期间不会持续保持网络连接，API 提交的内容会先保存到服务器，并在设备下一次自动唤醒后获取和显示。

## 固定内容和循环内容如何影响刷新

循环内容会按照列表顺序和设置的间隔切换。

固定内容会在指定时间窗口内生效。当进入或离开固定内容窗口时，设备可能会优先切换内容，而不是等到普通循环间隔结束。

如果设备同时遇到睡眠、固定内容切换、循环间隔等多个时间点，系统会综合这些候选时间点，选择更合适的刷新时机。

## 常见情景

### 情景 1：设备接入电源，并通过 API 立即显示内容

设备已接入电源，并且网络连接稳定。此时您通过文本 API 或图像 API 提交内容，并将 `refreshNow` 设置为 `true`。

在这种情况下，设备通常可以较快收到服务器下发的刷新任务，并切换到新的内容。它适合用于验证码、日程提醒、自动化脚本结果、桌面状态屏等对显示时机较敏感的场景。

### 情景 2：设备使用电池，并处于睡眠状态

设备使用电池供电时，会在不需要刷新内容时进入低功耗睡眠。此时您通过 API 提交了一段新的文本内容。

这段内容会先保存到服务器。设备不会因为这次 API 请求立即从低功耗睡眠中持续保持在线，而是在下一次自动唤醒后，重新连接 Wi-Fi 和服务器，再获取并显示最新内容。

因此，电池供电更适合展示固定内容、循环内容、天气、RSS、日历摘要等对显示时机不敏感的内容。

### 情景 3：设备每 30 分钟循环刷新一次

假设您设置的循环刷新间隔为 30 分钟，设备在 5:45 完成了一次内容刷新。

如果期间没有其他操作，下一次自动刷新大约会发生在 6:15。

如果您在 6:10 通过 App 手动切换了一次内容，或通过 API 立即切换了一次内容，那么这次 6:10 的刷新会成为新的时间基准。下一次自动刷新大约会发生在 6:40，而不是原本的 6:15。

### 情景 4：设备设置了固定内容时间段

假设您设置了一个固定内容，在每天 9:00 到 10:00 显示。设备当前正在显示循环内容，并且循环刷新间隔还没有结束。

当时间接近 9:00 时，固定内容的开始时间会成为一个更重要的刷新候选点。设备可能会优先切换到固定内容，而不是继续等待当前循环间隔结束。

当固定内容时间段结束后，设备也会重新计算后续刷新时间，并恢复到合适的非固定内容逻辑。

### 情景 5：多个设备在同一时间需要刷新

如果大量设备都设置为相同的刷新时间，服务器会在调度时保留必要的时间浮动，避免设备在同一时刻集中请求造成阻塞或超时。

因此，即使设备接入电源，自动刷新也不应被理解为严格到秒级的定时器。对于需要更接近准确时刻的内容，建议使用接入电源的设备，并通过 API 主动控制显示内容。

## API 使用建议

如果您的内容对显示时间比较敏感，例如验证码、日程提醒、自动化脚本结果或实时状态，建议：

1. 让设备持续接入电源；
2. 保持设备网络稳定；
3. 使用 API 设置内容；
4. 将 `refreshNow` 设置为 `true`。

如果设备使用电池供电，更建议展示固定内容、循环内容、天气、RSS、日历摘要等对显示时间不敏感的内容。
